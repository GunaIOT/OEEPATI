Kapan Data Masuk ke Database
Kejadian 1
Mesin Menyala & ESP32 Mengirim Data
Setiap sensor mendeteksi produk, ESP32 publish ke MQTT.
Backend menerima dan update counter di memori (belum ke DB).
MQTT: oee/machine1/count → "1" state.m1.inputOne++
Kejadian 2
Auto-Snapshot Setiap 5 Menit
Backend otomatis simpan kondisi mesin ke database setiap 5 menit.
Ini memastikan data tidak hilang walau tidak ada reset.
INSERT INTO machine_production
(machine_id, input_good, input_no_obj, oee_pct, ...)
VALUES (1, 540, 12, 64.27, ...)
machine_production ← INSERT tiap 5 menit otomatis
Kejadian 3
Tombol RESET Ditekan
Sebelum data dikosongkan, backend simpan snapshot akhir terlebih dahulu.
Ini memastikan data shift penuh tersimpan.
machine_production ← INSERT (snapshot akhir) lalu counter → 0
Kejadian 4
Tombol SUBMIT DATA Ditekan di Reject Packaging
Frontend kirim data via MQTT → backend terima → simpan ke 2 tabel sekaligus.
reject_submissions (header) + reject_details (per jenis).
INSERT INTO reject_submissions → id = 5
INSERT INTO reject_details (submission_id=5, 'Kosong', 3)
INSERT INTO reject_details (submission_id=5, 'Seal', 2)
reject_submissions ← INSERT reject_details ← INSERT (per jenis) dipicu user
Akses Data
Ambil Laporan via REST API
Dashboard atau tools lain bisa ambil data via API kapan saja.
GET /api/laporan?tanggal=2026-02-25&shift=1
GET /api/reject/per-jenis?tanggal=2026-02-25

Relasi Antar Tabel
DIAGRAM RELASI
machine_production — berdiri sendiri, tidak ada FK
└─ setiap baris = 1 snapshot mesin

reject_submissions — 1 baris per klik Submit
└─ 1 submission punya banyak detail → reject_details
└─ submission_id (FK) → reject_submissions.id
└─ ON DELETE CASCADE (hapus submission → detail ikut terhapus)

VIEWS (tidak perlu diisi manual):
v_daily_summary    ← GROUP BY dari machine_production
v_daily_reject     ← GROUP BY dari reject_submissions
v_reject_by_type   ← JOIN reject_details + reject_submissions
